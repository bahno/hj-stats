# hj-stats — Design (v1)

**Date:** 2026-07-06
**Status:** Approved design, pre-implementation
**Repo:** https://github.com/bahno/hj-stats (public) · deployed to GitHub Pages

## Context

hj-stats is a tool for analyzing **high jump** ranking statistics under the World
Athletics (WA) ranking system. The core need: understand how ranking points are
earned, and **visualize how the same performance yields different points across
competition categories** (OW, DF, GW, GL, A–F). This is useful because a jump of a
given height is worth very different ranking points depending on the competition's
category, and that difference is not obvious.

The project starts from an empty repo with a placeholder page already deployed to
GitHub Pages (deploy-from-branch). This spec covers **v1**; real-results analysis is
explicitly deferred to v2.

## Goal & scope (v1)

Build a static, fully-online web app with two features, both powered by one shared
ranking-points engine:

1. **Points Calculator (main screen)** — four simple selectors: **gender, height,
   competition category, finishing position** → the Ranking Score, broken down into
   Performance + Placing.
2. **Category Comparison** — for a given gender + height + position, visualize how the
   total score differs across competition categories (the headline feature).

Scoring is **single-result** for v1: `Result Score = Performance Score + Placing Score`.

### Out of scope (→ v2)
- Real-results analysis (World Athletics GraphQL scraper, athlete profiles).
- Best-N averaging of an athlete's ranking score over the scoring period.
- Qualifying-round placing scores (v1 uses the **final** placing table only).
- Championship/other bonuses and drop-off rules.

## The ranking-points model

Confirmed with the user:

```
Result Score = Performance Score + Placing Score
```

- **Performance Score** — a function of **gender + mark (height)**. Men's and women's
  high jump use different scoring tables. Exact values come from the official WA
  Scoring Tables. Encoded as gender → (mark → points) lookup.
- **Placing Score** — depends on **competition category** × **finish position**. Higher
  category and better position award more points. Encoded as category → {position → points}.

Competition categories: **OW, DF, GW, GL, A, B, C, D, E, F**.
Example confirmed placing values (1st place, T&F final): OW 375, DF 240, GW 200,
GL 170, A 140.

## Architecture

Static React + Vite SPA on GitHub Pages. All computation is client-side; there is no
runtime backend. Reference data is generated by a Python pipeline into committed,
reviewable JSON. Load is a non-issue (CDN-served static assets).

```
hj-stats/
├── pipeline/                      # Python — reference-data generation
│   ├── parse_scoring.py           # official Scoring Tables PDF → scoring_table.json
│   ├── scrape_placing.py          # ranking-rules HTML       → placing_points.json
│   ├── verify.py                  # asserts known example values; fails loudly
│   └── requirements.txt           # pdfplumber, requests, beautifulsoup4
├── src/
│   ├── data/                      # generated JSON (committed, reviewed via PR)
│   │   ├── scoring_table.json
│   │   ├── placing_points.json
│   │   └── categories.json
│   ├── engine/                    # pure TypeScript points model (unit-tested)
│   │   ├── score.ts               # performance/placing/result lookups
│   │   └── score.test.ts
│   ├── components/                # React UI
│   │   ├── Calculator.tsx
│   │   ├── CategoryComparison.tsx
│   │   └── charts/                # Recharts wrappers
│   ├── App.tsx
│   └── main.tsx
├── .github/workflows/
│   ├── deploy.yml                 # build Vite → deploy to Pages (on push to main)
│   └── refresh-data.yml           # weekly cron → run pipeline → PR if JSON changed
├── vite.config.ts                 # base: '/hj-stats/'
├── package.json
└── docs/superpowers/specs/2026-07-06-hj-stats-design.md
```

**Data flow:** Python pipeline parses the Scoring Tables PDF + scrapes the placing HTML
→ writes JSON to `src/data/` → committed → React imports JSON → TS engine computes
scores client-side → Recharts renders. The weekly Action re-runs the pipeline and
opens a PR only when a value actually changes.

## Data pipeline

**`parse_scoring.py`** — parses the official WA **Scoring Tables PDF** with `pdfplumber`,
extracts **both the men's and women's** high jump columns (mark ↔ points), emits:

```json
// scoring_table.json
{
  "event": "high_jump",
  "unit": "m",
  "source": "World Athletics Scoring Tables 2025",
  "points_by_mark": {
    "men":   { "2.30": 1244, "2.29": 1234, "...": 0 },
    "women": { "2.06": 1244, "2.05": 1233, "...": 0 }
  }
}
```

**`scrape_placing.py`** — fetches `worldathletics.org/world-ranking-rules/track-field-events-2025`
with `requests`, parses the **final** placing table for the relevant field-event group
with `beautifulsoup4`, emits:

```json
// placing_points.json
{
  "source": "World Athletics World Ranking Rules 2025 — Track & Field",
  "final": {
    "OW": { "1": 375, "2": 330, "...": 0 },
    "DF": { "1": 240, "...": 0 }
  }
}
```

```json
// categories.json — codes, human names, example competitions
[ { "code": "OW", "name": "Olympic Games / World Championships" }, "..." ]
```

**`verify.py`** — asserts a handful of known ground-truth values (e.g. a specific
height→points from the official table, and 1st-place placing points per category shown
above). Runs in the refresh Action; a mismatch fails the job so no bad PR opens.

**Assumption to resolve during implementation:** confirm which placing sub-table
(“Main Events” vs “Similar Events”) high jump belongs to in the WA rules, and whether
the Scoring Tables PDF lists high jump at 1 cm granularity (expected).

## Ranking-points engine (`src/engine/score.ts`)

Pure functions, no React, fully unit-tested:

- `performanceScore(gender, heightMeters): number` — exact lookup in
  `scoring_table.json`; returns `null`/throws for marks not in the table (input is
  constrained to valid marks).
- `placingScore(category, position): number` — lookup in `placing_points.json`;
  positions beyond the table award 0.
- `resultScore(gender, heightMeters, position, category): { performance, placing, total }`.
- `compareCategories(gender, heightMeters, position, categories[]): Array<{category, total, ...}>`
  — powers the comparison charts.

## Frontend

React + Vite + TypeScript SPA. One lightweight top toggle switches between two views.
Responsive (same layout collapses to a single full-width column on mobile).

**① Calculator (main screen)** — deliberately simple, a single centered card:

```
   hj-stats               [ Calculator | Compare ]
  ------------------------------------------------
              Gender     [ Men    v ]
              Height     [ 2.30 m v ]
              Category   [ OW     v ]
              Position   [ 1st    v ]
             --------------------------
              Ranking Score      1394
              Performance 1244 · Placing 150
```

- Inputs: **gender** toggle, **height** (valid marks only), **category** dropdown,
  **position** dropdown.
- Output: large tabular-figure **Ranking Score**, with the Performance + Placing
  breakdown small underneath.

**② Compare** — reuses the same selectors *except category* (category is the axis being
compared) and shows every category at once. Charts via **Recharts**:
  - **Bar chart** — total Ranking Score per category at the chosen gender + height + position.
  - **Line chart** — Ranking Score vs height, one line per category (fixed position),
    showing where categories diverge.
  - A **table** beneath the charts with exact numbers.

Charting library: **Recharts** (React-native ergonomics). ECharts is a fallback if a
future view needs heavier visualization.

### Visual design

Direction: **calm data dashboard**. Near-black / navy base, a single cool
**electric-blue** accent (active tab, chart bars/lines, the big score), restrained and
analytical. **Tabular (monospaced) numerals** for all scores so figures align cleanly.
Generous spacing, minimal chrome — the numbers are the interface. Consistent with the
dark placeholder already deployed.

## Deploy

- Switch GitHub Pages from **deploy-from-branch** to a **GitHub Actions build**
  (`deploy.yml`: `npm ci` → `vite build` → `upload-pages-artifact` → `deploy-pages`,
  on push to `main`).
- `vite.config.ts` sets `base: '/hj-stats/'` so assets resolve under the project
  subpath. Replace the placeholder `index.html` with the app entry.
- `refresh-data.yml`: weekly cron → set up Python → run pipeline → `verify.py` →
  open a PR (e.g. `peter-evans/create-pull-request`) only if `src/data/*.json` changed.

## Error handling

- Inputs constrained to valid marks/places/categories; out-of-range height or place
  shows a clear message — never a silent wrong answer.
- Engine functions fail explicitly on unknown marks rather than guessing.
- Pipeline fails loudly (`verify.py`) rather than shipping a garbled parse.

## Testing & verification

- **Pipeline:** `verify.py` asserts known height→points and place×category→points
  values against the official sources.
- **Engine:** Vitest unit tests for `performanceScore`, `placingScore`, `resultScore`,
  `compareCategories`, including out-of-range edge cases.
- **UI:** a light component test asserting the Calculator renders the correct breakdown
  for a known input.
- **End-to-end:** `npm run dev`, enter a height/place/category, confirm the breakdown
  matches a hand-checked value; open Comparison and confirm the bar/line charts and
  table agree with the engine. After deploy, load `https://bahno.github.io/hj-stats/`
  and repeat one spot-check.
